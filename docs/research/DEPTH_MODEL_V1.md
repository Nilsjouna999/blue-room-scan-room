## The shared spine (answer to Q2)
> **Serial = the join key. Archetype class = the voice-and-content rule. Both photo-derived;
> neither inferred about the person.**

- **SERIAL** (`objectNo → scanId → card.serial → mintSerial`) is the one **photo-grounded,
  person-neutral** join key every future surface cites. Verified: `scanId` derives from
  `objectNo` (data.js:473); the masked Free `····` placeholder (app.js:853) is literally the
  slot the revealed Halo serial fills (app.js:845) — so it **already carries the
  latent→developed physics.** Every surface (Codex, Journal/blog, Oracle, zodiac, numerology,
  MBTI, readings) is a **chapter ABOUT the record** that *cites the serial, never re-reads the
  person*. The serial only **receives and connects** lore; it never generates it.
- **ARCHETYPE CLASS** is the content axis: `class → family → lens pair → hidden-stat direction`
  governs **what each surface says** (its Codex entry, journal voice, oracle flavour, reading
  emphasis). The serial governs **how** surfaces connect; the class governs **what** they say.

## Link grammar (the rules)
1. **Direction is always inward** — every surface points TO the card record via serial
   (chapter → record), never record → inferred person. The serial is a *lookup key, not a
   trait key*, so inference is structurally impossible.
2. **Two-track attachment law** (the critical-law guard — how zodiac/numerology attach safely):
   - **Track A — READ (photo → card):** archetype class, oracle, aura, stat reads, scene role
     are derived from the image and labelled artifact evidence on the photo-read plates (01–05/07).
   - **Track B — FILE (declared → serial):** user-supplied zodiac / numerology / MBTI attach
     ONLY to the card's mint identity (serial + objectNo) as a **declared-lore rider on Plate 06
     (Mint Record)**, parallel to the production record.
   - **Forbidden cross:** no declared lore may appear on any photo-read plate; the Oracle may
     not absorb it; aura/archetype/stats may **never** be recalculated by it. Declared lore
     *files against the serial after Develop* — the system never computes it from the photo.
     (Canon: `GOLDEN_NUGGETS` #9, LOCKED via DECISION_LOG 2026-06-12 — *the photo creates the
     card; the profile adds the lore; never inferred.*)
3. **Class axis** — one-to-one within a card, one-to-many across surfaces (one class governs
   its Codex page, journal register, oracle flavour, reading emphasis). The class is the card's
   photo-derived identity — NOT where declared profile data attaches (that is Track B).
4. **Codex vs Journal** share the spine on **different index axes** so they are not the same
   app: Card → **Codex** is a **class** pointer (one page per class, every card of that class;
   class-level lore — discovery note + lens pair + hidden-stat direction — lives here). Card →
   **Journal** is a **date/serial-order** pointer (each card once, chronologically; links back
   only to the Card Record, never to the Codex). Codex = taxonomy; Journal = time.
5. **One landing surface** — all of it activates at the **Card Record back-seam (P4)** where
   Develop assigns the serial (serial reveal, oracle-timeline teaser, declared-lore rider,
   Codex/Vault seams). Never a new floating surface. Vault/Codex stay backlog: note the seam,
   do not build the room.

## Gloss system (answer to Q3)
**Two-tier hover with a hard glossable/mysterious boundary.**
- **Tier 1 (hover):** one line in receipt-chip grammar — `[term]: [what it observes in the
  image]` — drawn **verbatim from the doc that already defines it** (CARD_SYSTEM_V1 §2 stat
  meaning, §3 lens "Observes" column, CARD_LOGIC_V1 §2 tier band, COPY_SYSTEM §4 archetype
  one-liner). Always optional, one line, never blocks the read.
- **Tier 2 (optional, quiet link):** only when the destination is an **already-built grounded
  surface** (a Codex class page once it exists, the relevant dossier plate, the spec section);
  absent otherwise.
- **Destination constraint (Artifact Test on gloss copy):** a gloss may explain what a term
  **observes** in the photograph; it may **never** explain what it **implies about the person**
  (no "this means you are X").
- **Glossable iff** (a) it has a grounded image-artifact definition in the canon lexicon (stat
  labels, tier bands, lens names, archetype class names, zone names, receipt grammar) **AND**
  (b) it appears **outside the Oracle register**.
- **Left mysterious — no hover, no link:** Oracle / discovery-note fenced fiction, the "in
  another timeline" teaser, sealed-stat teaser copy. **The opacity IS the product**
  (CARD_LOGIC_V1 §11, LOCKED-grade). A term in both contexts is glossable in the non-Oracle
  context only. (This just runs the existing COPY_SYSTEM §3 preferred/banned lexicon split at
  *hover time* instead of authoring time — same law, new layer.)
